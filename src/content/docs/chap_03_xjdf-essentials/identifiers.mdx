The XJDF Specification defines several types of identifiers (IDs) in order to identify resources, jobs, devices etc. This section gives a brief introduction to the most important IDs, what they are responsible for, and their appropriate scope. An identifier has to be unique within its scope. For instance, the scope “Production” means the identifier has to be unique within the whole production system. A production system may cover several sites. The following table contains the most significant XJDF IDs:



| ID Name | Description | Scope | Example |
| ------- | ----------- | ----- | ------- |
| `ExternalID` | Unique identifiers of physical resources. | Production | `COMP-34` |  
| `ID` / `IDRef` | Unique identifiers and references within a single XJDF Document | Document | `j3hduw83` |
| `Partition Key` | Unique identifier of a specific partition. | Production | `SHEET-1` |
| `JobID` | Unique identifier of a job. | Production | `JB-124` |
| `JobPartID` | Identifies one or more worksteps of the same type | Production | `Front` |
| `DeviceID` | Unique identifier of a device. | Production | `PRESS-1` |


## ExternalID
The @ExternalID is “an ID of the resource as defined in the MIS system. For instance item codes or article numbers or identifiers on semi-finished products or Resource elements. @ExternalID shall be used to uniquely identify resources and products for the purpose of inventory tracking” [@xjdf_ip1, chap. 6 Table 6.1: Resource/ExternalID].
Generally, each resource can be assigned by an item code, article number, or identifiers. An ExternalID has to be unique within a whole production system. Once defined, the resource’s ExternalID should not be changed.

```xml title="Example: ExternalID in Resources"
[...]
<xjdf:ResourceSet Name="Component" Usage="Input">
    <xjdf:Resource ExternalID="COMP_ID_UNFOLDED">
        <xjdf:Component Dimensions="841.88 595.27 0.42"
            MediaRef="MEDIA-ID"/>
    </xjdf:Resource>
</xjdf:ResourceSet>
<xjdf:ResourceSet Name="Component" Usage="Output">
    <xjdf:Resource ExternalID="COMP_ID_FOLDED">
        <xjdf:Component Dimensions="280.69 595.27 1.26"
            MediaRef="MEDIA-ID"/>
    </xjdf:Resource>
</xjdf:ResourceSet>
[...]
```

This example is an extract of the Folding Example and shows two ResourceSet elements, each containing a Component Resource. Both Resource elements describe semi-finished goods. The upper one is the description of the unfolded sheet as input resource, whereas the lower one refers to the specification of the folded sheet as the output resource of the folding process. Both resources are identified by an individual `@ExternalID` attribute. The input resource is identified by `@ExternalID = "COMP_ID_UNFOLDED”` and the output resource is identified by `@ExternalID = "COM_ID_FOLDED"`. Both identifiers identify the appropriate resources during their lifecycle. During the folding process, the lifecycle of the component `COMP_ID_UNFOLDED` ends, while the lifecycle of `COMP_ID_FOLDED` begins.
In a real-world scenario, the ExternalID may be printed out on a label and stuck on the component’s pile for identification. This enables users to easily identify the individual components in the warehouse. 


## ID / IDRef
ID / IDRef attributes are for internal document references only. The values of these attributes may change from one transaction to the next. ID / IDRefs must not be used to identify resources within the production workflow. Valid locations for IDs are the attributes `//ResourceSet/@ID` and `//Resource/@ID`. 
“Unless otherwise specified, an `@IDREF` or `@IDREFS` will refer to an individual Resource rather than an entire ResourceSet” [@xjdf_ip1, chap. 3.3 "ResourceSet"].

```xml title="Example: Internal document reference using ID / IDRef"
[...]
<xjdf:ResourceSet Name="Component" Usage="Output">
    <xjdf:Resource ExternalID="COMP_ID_FOLDED">
        <xjdf:Component Dimensions="280.69 595.27 1.26" 
            MediaRef="MEDIA-ID" />
    </xjdf:Resource>
</xjdf:ResourceSet>
[...]
<xjdf:ResourceSet Name="Media">
    <xjdf:Resource ID="MEDIA-ID">
        <xjdf:Media MediaType="Paper" Thickness="150" Weight="135" 
            ISOPaperSubstrate="PS3" />
    </xjdf:Resource>
</xjdf:ResourceSet>
[...]
```

This example is also an extract of the Folding Example. The first ResourceSet contains a Component resource and the second one a Media resource. The Media resource is referenced by the Components `@MediaRef` attribute. The value of the attribute `//Component/@MediaRef` is equals to the ID value of the Media Resource (see `//ResourceSet/Resource/@ID`). This reference is used to obtain additional information about the paper of the component.

The `MEDIA-ID` is NOT the global identifier of the paper or pile in the company. The IDs value is for internal document references only and may change from one transaction to the next. A global media identifier would have to be written as the value of the @ExternalID attribute. 


## Partition Key
Whenever a ResourceSet contains several Resource elements which are NOT referenced using ID / IDRefs, the subselection of the appropriate resources is based on partitioning. Partitioning is an XJDF specific mechanism and is explained in the chapter “Resources”.
Partition keys are called the attributes of the Part element. In contrast to ExternalIDs and the ID / IDRefs references, the values of these attributes are normally given by the specification.

```xml title="Example: Selection by Partition Keys"
[...]
<ResourceSet Name="ExposedMedia" Usage="Input" ProcessUsage="Plate">
    <Resource>
        <Part Separation="Cyan"/>
        <ExposedMedia MediaRef="MEDIA-PLATE-ID" PlateType="Exposed" />
    </Resource>
    <Resource>
        <Part Separation="Magenta"/>
        <ExposedMedia MediaRef="MEDIA-PLATE-ID" PlateType="Exposed" />
    </Resource>
    <Resource>
        <Part Separation="Yellow"/>
        <ExposedMedia MediaRef="MEDIA-PLATE-ID" PlateType="Exposed" />
    </Resource>
    <Resource>
        <Part Separation="Black"/>
        <ExposedMedia MediaRef="MEDIA-PLATE-ID" PlateType="Exposed" />
    </Resource>
</ResourceSet>
[...]
```

The example above has already been shown in the Resources chapter and outlines an ResourceSet ExposedMedia containing four different kinds of ExposedMedia resources. All four items are distinguished by their Partition Key `@Separation` (Cyan, Magenta, Yellow and Black). 

The values of the partition key @Separation are given by the specification:

## JobID, JobPartID, DeviceID...
Apart from the identifiers mentioned above, XJDF comes with a huge bunch of identifiers for many different applications. The usages of these IDs are explained in the specification and do not require additional ex planations. The IDs - JobID, JobPartID, and the DeviceID - which are frequently used in XJDF, are explained in the following paragraph.

The JobID is the “Job identification used by the application that created the XJDF Job. Typically, a Job is identified by the internal order number of the MIS system that created the Job” [@xjdf_ip1, chap. 3 Table 3.1: XJDF/JobID]. An example value would be “674R834”. A JobID has to be unique within a production workflow and should not be changed once it has been defined.
The ”JobPartID SHALL identify one or more worksteps of the same type that can be described as one XJDF. `@JobPartID` is internal to the MIS system that created the XJDF” [@xjdf_ip1, chap. 3 Table 3.1: XJDF/JobPartID]. Example values are “Front” and “Back”.
The DeviceID is the “identifier of the Device. The @DeviceID SHALL be unique within the workflow. `@DeviceID` SHALL be the same over time for a specific Device instance (i.e., SHALL survive reboots)” [@xjdf_ip1, chap. 6 Table 6.58: Device/DeviceID]. A example value would be “SM-162-8”.

The explanations of all three identifiers are successively taken from the XJDF Specification. The specification defines the exact purpose of an ID, its data type, as well as the scope of the identifier. “The word ‘Unique’ without further scope details means ‘Unique within the job, message or file depending on the context’” [@xjdf_ip1, chap. 1 Table 1.4: Glossay (Unique)].


## Measurements, Units, and Data Types
“XJDF specifies most values in default units. That means that an implementation SHALL use the defined default units and SHALL NOT use alternate units [...]. Processors SHOULD NOT specify a unit unless no default exists, such as when new resources are defined. Then the units SHALL be based on metric units. Overriding the default units that are defined [...] is non-standard and MAY lead to undefined behavior. Any exceptions are specified in the appropriate descriptive tables” [@xjdf_ip1, chap. 1.9.1 "Units"]. All default units, their representation, and related remarks can be found in the XJDF Specification in chapter 1.9.1, Units.

The most significant measurement in XJDF is the length. With the exception of microscopic lengths, all length values are defined in desktop publishing points (dtp points). Examples are width, height, dimensions, positions, etc. A point is defined as 1/72nd of an inch. The following gives instructions on the conversion of millimeters to points and vice versa:

```java title="Conversion dtp points (dtp) to millimeter (mm) and vice versa"
// Conversion dtp points (dtp) to millimeter (mm)
mm = dtp * 72.0 / 25.4 

// Conversion millimeter (mm) to dtp points (dtp)
dtp = mm * 25.4 / 72.0 
```

Length values are used in various data structures, depending on the particular case they are used in. This table is an excerpt of common XJDF data types which are relevant with respect to  length expressions:

| Data Type | Description | Example |
| --------- | ----------- | ------- |
| `float` | A float is a single value expression as used for width or height expressions. | `"595.27"` |
| `rectangle` | Attributes of type rectangle are used to describe rectangular locations on the page, Sheet or other printable surface. A rectangle is represented as an array of four float numbers - llx lly urx ury - specifying the lower-left x, lower-left y, upper-right x and upper-right y coordinates of the rectangle, in that order. The origin for positioning is always the lower left corner of the parent object. | `"10.0 10.0 290.69 605.27"` |
| `shape` | A shape is represented as an array of three (positive or zero) float numbers - x y z - specifying the width x, height y and depth z coordinates of the shape, in that order. A shape is typically used for three-dimensional objects such as a book block. | `"280.69 595.27 1.26"` |
| `XYPair` | Attributes of type XYPair are mainly used to describe sizes like @Dimensions [...]. They can also be used to describe positions on a page. An XYPair is typically used for two-dimensional definitions such as page size. | `"290.69 605.27"` |


The table is an excerpt of data types as defined in the XJDF Specification whose relevance pertains to lengths. A data type is defined for each attribute in the specification. The following is an extract of the Folding example which contains some length expressions:

## Folding Schemes and Pagination
Folding and pagination are major tasks in a print production. Folding describes how a sheet or a final product has to be folded (e. g. zig-zag fold) and pagination defines how the single pages have to be ordered and positioned. Both are closely related, as folding schemes have a major effect on the positioning and orientation of pages on a sheet. 

In XJDF, each print product is described as a set of one or multiple product parts. A simple flyer or a folded leaflet consists of just one single product part, whereas a complex product consists of at least two or even more product parts. The following drawing outlines the product parts of a brochure containing 24 pages with two different types of papers - one for the cover and one for the body. Therefore, the brochure consists of the two product parts “Cover” and “Body”. The cover has four pages and is printed on premium paper, whereas the body consists of 20 pages and is printed on a more common type of paper:

<figure>
  <Image src={diagramBrochureProductParts} alt="Product parts of a softcover-brochure" />
  <figcaption>Figure: Product parts of a softcover-brochure</figcaption>
</figure>

In order to produce the desired print product, each product part has to be separated into one or several bindery signatures. “A BinderySignature represents both sides of a folding signature, a die cut surface or a flat product such as a postcard, each with one or more pages” [@xjdf_ip1, chap. 6.6 "BinderySignature"]. Each bindery signature is defined by a specific folding scheme. The exact distribution of the product parts in bindery signatures depends on the machinery of the appropriate printing house and the exact production plan of how the brochure is intended to be produced. 

Note: The bindery signature is only the logical description of how to fold and how to position the single pages on a component. Each logical bindery signature is related to a physical component in the production process. Components specify “the various versions of semi-finished goods in the press and postpress area, such as a pile of folded sheets that have been collected and are then be joined and trimmed” [@xjdf_ip1, chap. 6.18 "Component"]. 

A brochure with sixteen pages and only one product part can be separated into either a 16-page bindery signature, two 8-page bindery signatures, or into four 4-page bindery signatures. Combinations thereof are also permissible. The production plan is automatically calculated by the Management Information System, or manually by a production operator for the purpose of determining the most efficient production plan. 
The brochure in the example above may be separated into three bindery signatures. A 4-page bindery signature for the cover, and both a 16-page and a 4-page bindery signature for the body. The exact type of the bindery signatures is defined by the `@FoldCatalog` identifier. More about the `@FoldCatalog` identifiers and their respective meanings later in this chapter. The following is a drawing which outlines the bindery signatures and their corresponding folding scheme for the example brochure:

  <figcaption>Figure: Example of BinderySignatures for a softcover-brochure</figcaption>
</figure>

The first bindery signature on the left (gray) represents the cover of the brochure. This bindery signature has four pages and contains the first and the last two pages. The largest bindery signature is the second one. This one contains sixteen pages of the body (pages 3 to 10 and 15 to 22). The pages are positioned in two rows and four columns. Because of folding, all pages in the lower row are rotated by 180 degrees. The third bindery signature on the right has four pages and contains the center pages (pages 11 to 14).
Each component, described by a BinderySignature element, is printed and folded individually. In order to produce the final product, the three components are assembled and stitched together:

<figure>
  <Image src={diagramBrochureGathering} alt="Collecting of Components for a Softcover-Brochure" />
  <figcaption>Figure: Collecting of Components for a Softcover-Brochure</figcaption>

The distribution, positioning and orientation of the single pages on the different components is crucial. Pagination determines “the distribution and orientation of pages on a BinderySignature [...]" [@xjdf_ip1, chap. 5.4.22.1 "Pagination in Stripping"].
The pagination depends on the finishing process, that is, the planning for how the final product will be produced. It depends on the number, size, and types of bindery signatures, as well as on how the individual bindery signatures are supposed to be bound (collecting or gathering). Collecting means that the bindery signatures are collected on a spine and are placed within one another. The first bindery signature is on the outside of the final product (as shown in the example). Gathering is the alternative method of assembling. When gathering, all bindery signatures are gathered on a pile and placed on top of one another so that the first one is on the top of the final product. This method is used for non stitching binding types.

A bindery signature is conceptually the smallest physical component in which all print products have to be separated. Even a simple two-sided flyer has to be described as an F2-1 bindery signature. Therefore, bindery signatures play a major role in a print production workflow. At a minimum, a bindery signature is defined by its FoldCatalog attribute value and the partition key `@BinderySignatureID`. 

```xml title="Example: A BinderySignature ResourceSet"
[...]
<xjdf:ResourceSet Name="BinderySignature" Usage="Input">
    <xjdf:Resource>
        <xjdf:Part BinderySignatureID="BS-1" ProductPart="COVER-ID" />
        <xjdf:BinderySignature FoldCatalog="F4-1"/>
    </xjdf:Resource>
    <xjdf:Resource>
        <xjdf:Part BinderySignatureID="BS-2" ProductPart="BODY-ID" />
        <xjdf:BinderySignature FoldCatalog="F16-7"/>
    </xjdf:Resource>
    <xjdf:Resource>
        <xjdf:Part BinderySignatureID="BS-3" ProductPart="BODY-ID" />
        <xjdf:BinderySignature FoldCatalog="F4-1"/>
    </xjdf:Resource>
</xjdf:ResourceSet>
[...]
```
This snippet is an extract of the XJDF Collecting example to be presented later and can be analyzed in detail at that point. The relevant details here are the `@BinderySignatureID` partition key, the `@FoldCatalog` attribute, and the XML element structure.
The `@BinderySignatureID` partition key is the “master identifier of a BinderySignature” [@xjdf_ip1, chap. 6 Table 6.6 Part/BinderySignatureID] within the production workflow. The value of the `@FoldCatalog` attribute is a fold catalog identifier which refers to a predefined standard fold scheme in the XJDF fold catalog. The format of the fold catalog identifier is “‘Fn-i’ where ‘n’ is the number of finished pages and ‘i’ is either an integer, which identifies a particular fold or the letter ‘X’ which identifies a generic fold” [@xjdf_ip1, chap. A.3.3 "Fold Catalogs"].

Arrow up (⬆) means the left section has to be folded over the right section or the bottom section has to be folded over the top section. Arrow down (⬇) indicates that the left section has to be folded under the right section or the bottom section has to be folded under the top section. The size of the appropriate section is expressed by a fraction of the open sheet size. Fractions of horizontal folds refer to the width of the open sheet, while vertical folds refer to the height. The first fold is always a horizontal fold. The plus sign ( + ) is used to toggle between the horizontal and the vertical folds.
The XJDF fold catalog provides folding schemes for the most standard situations in a print production workflow. In the event that there is no match, XJDF provides generic fold schemes. An example of a generic 18-pages bindery signature would be “F18-X”. 
Custom folds are also supported. The procedure for extending the XJDF by a custom folding scheme is explained in the chapter “Extensibility”. However, the disadvantage of both methods is that they pose limitations on automation. 

### XJDF Example: Collecting
Many collecting devices do not only collect pre-folded components, but also provide stitching and trimming. The input of such machines is a set of pre-folded bindery signature components, while the output is a finished brochure.

From the XJDF perspective, such devices are described by a Gray Box covering the processes Collecting, Stitching, and Trimming. The process Collecting “collects folded sheets or partial products[...]. The first component to enter the workflow lies at the bottom of the pile collected on a saddle, and the sequence of the input components that follows depends upon the produced component” [@xjdf_ip1, chap. 5.6.7 "Collecting"]. The process Stitching defines how “gathered or collected sheets or signatures are stitched together” [@xjdf_ip1, chap. 5.6.33 "Stitching"]. After collecting and stitching, the semi-finished magazines have to be trimmed on the head, the foot, and the face. This is done by the Trimming process. “The Trimming process is performed to adjust a book block or sheet to its final size” [@xjdf_ip1, chap. 5.6.37 "Trimming"].
The following example illustrates an XJDF Document describing a “Collecting Stitching Trimming” Gray Box:

```xml title="Example: XJDF Gray Box for a Collecting Device"
<xjdf:XJDF JobID="JOB-1234" Types="Collecting Stitching Trimming"
        xmlns:xjdf="http://www.CIP4.org/JDFSchema_2_0">
    <xjdf:AuditPool>
        <xjdf:AuditCreated>
            <xjdf:Header AgentName="MIS" AgentVersion="1.0"
                Time="2017-04-21T22:06:47Z" />
        </xjdf:AuditCreated>
    </xjdf:AuditPool>
    <xjdf:ProductList>
        <xjdf:Product Amount="1000" DescriptiveName="My Magazine A4" 
                IsRoot="true">
            <xjdf:Intent Name="BindingIntent">
```

The collecting example is quite similar to the folding example. One difference here is that the ProductList describes a complex product as opposed to a simple one. The complex product consists of a root product and two child products. The root product describes the overall brochure, whereas the two child products describe both product parts of the brochure - cover and body. More about complex products will be explained in the next chapter.
Since a collecting device provides more processes than a simple folding machine, the number of input resources is higher. The primary resources of interest here are Assembly and BinderySignature. As defined, the magazine has been separated into three bindery signatures. A four-page bindery signature for the cover and two bindery signatures (16 and 4 pages respectively) for the body. Using the `@ProductPart` partition key, each BinderySignature element references a child product. The Assembly element defines how the single bindery signatures have to be assembled in order to obtain the final product. The attribute `@BinderySignatureIDs` refers to the bindery signatures to be assembled. The order of the IDs within the attribute corresponds to the order in which all bindery signatures are to be assembled. The attribute `@Order` = “Collecting” defines the first ID mentioned and is the outer component, while the following ones are bonded within one another (see the figure “Collecting of BinderySignatures for a softcover-brochure”).
Each BinderySignature element of the magazine is linked to a Component using the `@BinderySignatureID` partition key. A component is the physical representation of a bindery signature.
StitchingParams and TrimmingParams are specific input resources for the processes Stitching and Trimming. In this example, StitchingParams defines an eyelet staple on the saddle of the magazine. TrimmingParams defines the final size of the magazine. The rest of the collecting example is quite similar to the folding example.
